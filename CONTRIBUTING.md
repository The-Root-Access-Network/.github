## How Can I Contribute?

### 🐛 Reporting Bugs
Found something broken? Open an issue using the **Bug Report** template. Please include:
- A clear title and description
- Steps to reproduce the issue
- Expected vs actual behaviour
- Screenshots if applicable
- Your environment (OS, browser, Node version, etc.)

### 💡 Suggesting Features or Improvements
Have an idea? Open an issue using the **Feature Request** template. We'd love to hear from you — especially ideas that help us reach underserved communities better.

### 👨‍💻 Code Contributions
Ready to write code? See [Getting Started](#getting-started) below.

### 📝 Documentation
Clear documentation is just as important as clean code. Fixing typos, improving explanations, and adding examples are always welcome.

### 🌍 Translations
If you can help translate educational content or UI into other languages spoken in our communities, please reach out via info@therootaccessnetwork.com.

## Getting Started

1. **Fork** the repository to your own GitHub account.
2. **Clone** your fork locally:
   ```bash
   git clone https://github.com/YOUR-USERNAME/REPO-NAME.git
   cd REPO-NAME
   ```
3. **Create a new branch** for your work (see branch naming below):
   ```bash
   git checkout -b feature/your-feature-name
   ```
4. **Install dependencies** (check the repo's README for specific instructions).
5. **Make your changes** — write clean, readable code with comments where helpful.
6. **Test your changes** — ensure existing tests pass and add new ones if needed.
7. **Commit your changes** (see commit guidelines below).
8. **Push** your branch:
   ```bash
   git push origin feature/your-feature-name
   ```
9. **Open a Pull Request** against the `main` branch of the original repo.

---

## Branch Naming Convention

Use lowercase and hyphens. Follow this format:

| Type | Format | Example |
|------|--------|---------|
| New feature | `feature/short-description` | `feature/add-quiz-module` |
| Bug fix | `fix/short-description` | `fix/login-redirect-error` |
| Documentation | `docs/short-description` | `docs/update-readme` |
| Refactor | `refactor/short-description` | `refactor/cleanup-api-calls` |
| Hotfix | `hotfix/short-description` | `hotfix/broken-contact-form` |

## Commit Message Guidelines

We follow a simplified version of [Conventional Commits](https://www.conventionalcommits.org/):

```
<type>: <short summary in present tense>
```

**Types:**
- `feat` – a new feature
- `fix` – a bug fix
- `docs` – documentation changes only
- `style` – formatting, missing semicolons, etc. (no logic changes)
- `refactor` – code restructuring (no feature/fix)
- `test` – adding or updating tests
- `chore` – maintenance tasks, dependency updates

**Examples:**
```
feat: add data support request form
fix: resolve broken nav link on mobile
docs: update installation steps in README
```

## Pull Request Process

1. Ensure your PR targets the `main` branch (unless directed otherwise).
2. Fill out the PR template completely — description, changes made, how to test.
3. Link any related issues using `Closes #issue-number`.
4. Request a review from a maintainer.
5. Be open to feedback — we review with kindness and the project's goals in mind.
6. Once approved, a maintainer will merge your PR.

**PR titles** should follow the same convention as commit messages.

## Reporting Bugs

Before filing a bug report:
- Search existing issues to avoid duplicates.
- Try to reproduce the issue on the latest version.

When filing, use the Bug Report issue template and be as detailed as possible.

---

## Suggesting Features

We welcome ideas that align with our mission: making cybersecurity education accessible. When suggesting a feature:
- Explain the problem it solves.
- Describe who benefits (students, mentors, community members, etc.).
- If possible, suggest how it might work.
